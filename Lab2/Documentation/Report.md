# Звіт до Lab2: «Калькулятор» за архітектурою MVC

## 1) Переваги та недоліки MVC у цьому застосунку

### Переваги

1. **Розділення відповідальностей**
   - `Model` відповідає за обчислення і стан.
   - `View` відповідає лише за відображення та генерацію подій.
   - `Controller` керує сценаріями взаємодії.

2. **Краща підтримуваність**
   - Зміни в інтерфейсі (кнопки, розкладка, стиль) не потребують переписування обчислень.
   - Зміни в обчисленнях не зачіпають код форми.

3. **Тестованість логіки**
   - Логіку `CalculatorModel` можна тестувати ізольовано, без запуску UI.

4. **Масштабованість**
   - Легко додавати нові операції (наприклад `%`, `sqrt`, пам’ять) через розширення `Model` і прокидання подій через `Controller`.

### Недоліки

1. **Більше коду та файлів**
   - Для простого калькулятора MVC створює додатковий «каркас» (інтерфейси, контролер, шари).

2. **Початковий поріг входу**
   - Новачкам складніше одразу зрозуміти потік подій між трьома шарами.

3. **Потрібна дисципліна**
   - Якщо порушувати ролі шарів (наприклад, логіку в UI), переваги MVC швидко зникають.

## 2) Порівняння з альтернативними підходами

### a) Монолітний підхід (усе у `Form`)
- **Плюс:** швидка реалізація маленького прототипу.
- **Мінус:** змішуються UI і бізнес-логіка, складніше вносити зміни, гірше тестувати.

### b) MVP (Model-View-Presenter)
- **Плюс:** Presenter може бути ще простіше тестувати, бо View часто максимально «пасивна».
- **Мінус:** часто більше шаблонного коду для зв’язків.

### c) MVVM (частіше WPF/MAUI)
- **Плюс:** сильна прив’язка даних (binding), менше ручних обробників подій.
- **Мінус:** для WinForms менш природно, додаткові абстракції для простого кейсу можуть бути надмірними.

**Висновок порівняння:** для WinForms-калькулятора MVC — збалансований компроміс між структурованістю та складністю.

## 3) Повторне використання рішень в інших проєктах

1. **Повторне використання `Model`**
   - `CalculatorModel` можна застосувати в іншому UI: WPF, MAUI, Web (через API), консольний клієнт.

2. **Повторне використання контракту `ICalculatorView`**
   - Дає змогу швидко підмінити інтерфейс без зміни контролера.

3. **Повторне використання підходу подій**
   - Шаблон «UI-подія → Controller → Model → оновлення View» підходить і для інших desktop CRUD/utility-застосунків.

## 4) Можливість внесення змін у застосунок

### Зміни, які вносяться легко

- Додавання нової операції:
  1. Додати значення в `OperationType`.
  2. Розширити обчислення в `CalculatorModel`.
  3. Додати кнопку у `CalculatorForm`.
  4. Прокинути подію через `CalculatorController`.

- Заміна UI (інша форма, інший фреймворк) без переписування ядра логіки.

### Потенційні точки ускладнення

- Якщо логіка стану стане значно складнішою (історія операцій, науковий режим), доцільно виділити додаткові сервіси або стейт-машину в `Model`.

---

## Загальний висновок

Реалізація калькулятора через MVC показує, що навіть простий desktop-застосунок можна зробити структурованим, розширюваним і більш придатним до супроводу. Архітектурна дисципліна дає вигоду не стільки в стартовій швидкості, скільки в подальших змінах, повторному використанні та якості коду.
