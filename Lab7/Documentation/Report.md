# Лабораторна робота №7
## Тема: «Текстовий редактор»

## 1. Опис застосунку
Розроблено GUI-застосунок **Lab7 Text Editor** на WinForms, що підтримує базові операції з текстом:
- створення нового документа;
- відкриття та збереження файлів;
- збереження в новий файл;
- редагування через Undo/Redo/Cut/Copy/Paste;
- взаємодію через меню, панель інструментів і статусний рядок.

## 2. Архітектурний підхід
Застосовано шарувану архітектуру з чітким поділом відповідальностей:
- **Domain**: бізнес-сутність документа (`TextDocument`).
- **Application**: сценарії роботи редактора (команди, контексти, абстракції).
- **Infrastructure**: технічні деталі (читання/запис файлів, фабрика обробників форматів).
- **Presentation**: WinForms-інтерфейс (`MainForm`).

Такий підхід спрощує тестування, супровід і розширення (наприклад, додавання нових форматів або команд).

## 3. SOLID у проєкті
1. **S — Single Responsibility Principle**:
   - `MainForm` відповідає за UI.
   - `FileService` — лише за файлові операції.
   - Кожна команда (`OpenDocumentCommand`, `SaveDocumentCommand` тощо) реалізує окрему дію.

2. **O — Open/Closed Principle**:
   - Нові формати файлів додаються через новий `IFileHandler` без зміни існуючого коду.
   - Нові дії додаються через нову команду, не ламаючи поточну структуру.

3. **L — Liskov Substitution Principle**:
   - Реалізації `IFileHandler` (`TextFileHandler`, `MarkdownFileHandler`) взаємозамінні.

4. **I — Interface Segregation Principle**:
   - Використані вузькі інтерфейси: `IFileService`, `IFileHandlerFactory`, `IEditorCommand`.

5. **D — Dependency Inversion Principle**:
   - Вищі рівні залежать від абстракцій (`IFileService`, `IEditorCommandFactory`), а не від конкретних реалізацій.

## 4. Використані патерни
### 4.1. Command (поведінковий)
Кожна дія користувача інкапсульована в окремому класі команди:
- `NewDocumentCommand`
- `OpenDocumentCommand`
- `SaveDocumentCommand`
- `CutCommand`, `CopyCommand`, `PasteCommand`

Переваги:
- легке додавання нових дій;
- зручне підключення команд до меню/toolbar;
- ізоляція бізнес-логіки від UI.

### 4.2. Factory Method / Simple Factory (породжувальний)
`EditorCommandFactory` створює команди за назвою, а `FileHandlerFactory` повертає обробник за розширенням.

Переваги:
- централізоване створення залежностей;
- менша зв’язаність між UI та конкретними класами.

### 4.3. Strategy (поведінковий)
Різні реалізації `IFileHandler` задають стратегію роботи з конкретним форматом файлу.

Переваги:
- легке розширення підтримки форматів;
- прозорий вибір алгоритму через фабрику.

### 4.4. Observer (поведінковий)
`DocumentSession` генерує подію `Changed`, а UI підписується на неї і оновлює статус.

Переваги:
- реактивне оновлення інтерфейсу;
- менша залежність між джерелом змін і візуалізацією.

## 5. Аргументи на користь обраної архітектури
1. **Масштабованість**: легко додати нові формати файлів, команди, сервіси.
2. **Підтримуваність**: код розділено по сенсових шарах і папках.
3. **Тестованість**: Application/Domain шари відокремлені від WinForms UI.
4. **Гнучкість**: залежності інвертовані через інтерфейси.

## 6. Висновок
У межах Lab7 реалізовано повноцінний базовий текстовий редактор із GUI, меню та панеллю інструментів. Архітектурно застосунок побудований за принципами SOLID і містить породжувальні та поведінкові патерни, що робить рішення практичним для подальшого розширення.
