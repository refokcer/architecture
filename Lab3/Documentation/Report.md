# Звіт до Lab3: "Калькулятор" у архітектурі Client-Server

## 1) Переваги та недоліки Client-Server для калькулятора

### Переваги
1. **Розділення відповідальностей**: UI (клієнт) і обчислення (сервер) ізольовані, що спрощує підтримку.
2. **Повторне використання API**: той самий серверний endpoint можна використати для вебклієнта, мобільного застосунку або desktop-клієнта.
3. **Централізація логіки**: правила обчислень і валідації зосереджені в одному місці (`CalculationService`).
4. **Масштабованість**: сервер можна масштабувати незалежно від клієнта.

### Недоліки
1. **Мережева залежність**: без доступу до сервера клієнт не виконає обчислення.
2. **Додаткова затримка**: навіть проста операція потребує HTTP-запиту.
3. **Складність інфраструктури**: потрібно розгортати і підтримувати серверну частину.

## 2) Порівняння з альтернативними архітектурами

1. **Монолітний desktop-калькулятор (без мережі, як WinForms)**:
   - Плюс: швидко, просто, офлайн.
   - Мінус: складніше повторно використовувати логіку в інших клієнтах.

2. **SPA з обчисленнями лише на клієнті (Pure Client-Side)**:
   - Плюс: не потрібен сервер для обчислень.
   - Мінус: логіка розпорошується по фронтенду, складніше централізовано контролювати правила/аудит.

3. **Microservices**:
   - Плюс: гнучка масштабованість, незалежний деплой.
   - Мінус: надлишково для простої задачі калькулятора.

## 3) Висновки щодо повторного використання рішень

1. **Серверний сервіс обчислень** можна використати в інших проєктах як окремий модуль.
2. **Контракти API (DTO)** придатні для генерації клієнтів і контрактного тестування.
3. **Патерн ізоляції логіки в сервісі** масштабовано переноситься на складні домени (фінанси, аналітика, звітність).

## 4) Можливість внесення змін у застосунок

1. Додавання нової операції (наприклад, `%` або `sqrt`) потребує:
   - розширити `OperationType`;
   - додати реалізацію в `CalculationService`;
   - додати елемент у клієнтський `select`.
2. Можна легко додати:
   - журнал історії обчислень;
   - авторизацію API;
   - логування і метрики;
   - unit/integration тести.
3. Завдяки розділенню на клієнт і сервер зміни в UI не ламають доменну логіку і навпаки.

## Підсумок

Архітектура Client-Server для калькулятора показує правильну інженерну декомпозицію навіть на простому прикладі: клієнт відповідає за UX, сервер — за бізнес-логіку. Це створює основу для масштабування, повторного використання коду і контрольованого внесення змін.
